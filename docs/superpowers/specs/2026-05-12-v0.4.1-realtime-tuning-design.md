## 1. Context & Motivation

사용자 피드백: "가격은 얼마 주기로 불러오고있어? 최대한 실시간 동기화하면 좋겠는데"

현재 (`v0.4.0`):

| 단계 | 주기 | 위치 |
|---|---|---|
| SWR client refresh | 5초 | `src/hooks/usePrices.ts:17` |
| SWR dedup | 4초 | `src/hooks/usePrices.ts:19` |
| Server singleFlight cache | 4초 (모든 source 통합) | `server/index.ts:90` |
| HTTP Cache-Control | `s-maxage=4, stale-while-revalidate=10` | `server/index.ts:114` |
| Naver 권장 polling | 7초 (`pollingInterval: 7000` in response) | Naver API |

**핵심 병목**: server singleFlight 가 4s 마다 모든 source 를 동시 fetch → Naver 7초 권장 위반 (이미 운영 중) + 다른 source (HL/Binance/Upbit) 가 1초 이하 실시간 가능한데 4초 cache 에 묶임.

### Goals

1. **A — HL WebSocket 활성화**: `HL_USE_WS=true` 환경변수 + `allMids` subscription → HL 가격 1초 미만 실시간.
2. **B — SWR client polling 5s → 2s**: 클라이언트 빠른 폴링 (server cache hit 으로 보호).
3. **C — Source 별 독립 cache TTL**: HL=1s / Binance=1s / Upbit=2s / Yahoo=5s / Naver=장중 2s, 휴장 7s. 단일 4s singleFlight → source 별 cache.
4. **D — Naver 장중 2s / 휴장 7s**: KRX 영업시간만 적극 단축, Naver 호출량 분산.

### Non-Goals

- 새 source 추가 ❌
- 가격 표시 UI 변경 ❌ (v0.4.0 그대로)
- Hyperliquid 외 다른 거래소 WebSocket ❌
- 클라이언트 측 별도 setInterval ❌ (SWR 만 사용)
- Realtime push (SSE / WS to client) ❌ (별도 v2 backlog)

### 카파시 / AI slop 가드

- Source 별 cache 는 단일 `sourceCache` Map (key=source name) 으로 통합. 별도 클래스/팩토리 X.
- TTL 값은 상수 객체 (`SOURCE_TTL_MS`) — 하드코딩 매직 넘버 X.
- Naver 동적 TTL 은 `getSessionState().krx` 한 줄 분기. 별도 helper 추출 X.
- HL WS 인프라는 이미 존재 (`server/lib/sources/hyperliquid-ws.ts`) — 신규 작성 X.

## 2. 데이터 흐름 변경

### 2.1 Source 별 TTL (C)

```ts
// server/lib/cache.ts 또는 신규 server/lib/source-cache.ts
const SOURCE_TTL_MS = {
  hyperliquid: 1000,
  binance: 1000,
  upbit: 2000,
  yahoo: 5000,
  // naver: 동적 — 장중 2s / 휴장 7s (D)
} as const;

function naverTtl(): number {
  return getSessionState().krx ? 2000 : 7000;
}
```

기존 `singleFlight('prices', 4000, ...)` 한 묶음 → source별 5개 `singleFlight` 호출 (각자 TTL).

### 2.3 HL WS (A)

코드 변경 없음 — 기존 `server/lib/sources/hyperliquid-ws.ts` 가 env 분기 처리. 단, **default 를 true 로 바꿀지** 결정 옵션 (현재 default false).

→ **결정: default false 유지**, env 명시적 설정만. 사용자 환경 차이 (개발 vs 프로덕션) 존중.

**HL WS asOf 동작 (Codex 🚨 #H 반영)**:
- `state.lastUpdate = Date.now()` 매 WS message 수신 시 갱신 (`hyperliquid-ws.ts:68`)
- `server/index.ts:108`: `payload.hl.asOf = ws.lastUpdate` — HL PricePoint 의 asOf 를 WS 마지막 message 시점으로 덮어씀
- 신선도: WS 가 정상 연결 시 mid 변경마다 push → asOf 가 1초 미만 신선 보장
- WS 끊김 (`state.connected === false`) 시: server/index.ts:102 가 WS 데이터 무시 → REST fetch 결과 (`payload.hl.asOf = REST fetch 시점`) 사용 → fallback 자동
- skew guard (§2.5) 는 이 asOf 값을 그대로 사용. HL WS 활성화 후에도 동일 작동

### 2.4 Naver 동적 TTL (D)

`getSessionState()` 호출은 가벼움 (timezone + holiday lookup) — 매 요청 호출 OK.

### 2.5 Timestamp Skew Guard (Codex 🚨 #1 반영)

Source 별 TTL 분리 시 HL 1s + Naver 7s 조합으로 두 가격의 fetch 시점 차이가 최대 ~6초까지 벌어질 수 있다. 그 격차로 premium 계산 시 stale-mix 위험.

```ts
// server/lib/normalize.ts 또는 assemble.ts 내
const MAX_PREMIUM_SKEW_MS = 5000; // 5초

function computePremiumWithSkew(
  hl: PricePoint | undefined,
  naver: PricePoint | undefined,
  fx: FxRates
): Premium {
  if (!hl || !naver) return { pctUsd: null, pctUsdt: null, guard: 'normal' };
  const skewMs = Math.abs(hl.asOf - naver.asOf);
  if (skewMs > MAX_PREMIUM_SKEW_MS) {
    // 시점 격차 너무 크면 premium null + guard='warn' 표시
    return { pctUsd: null, pctUsdt: null, guard: 'warn' };
  }
  // 기존 computePremium 로직 그대로
  return computePremium(hl.price, naver.price, fx);
}
```

5초 임계값 근거: 장중 (HL 1s + Naver 2s = max 3s skew) 정상, 휴장 (HL 1s + Naver 7s = max 8s skew) 시 premium 자동 N/A. 사용자 결정 시 가능 — 현재 안 = 5초 (장중 정상, 휴장 N/A) — 휴장 시 premium 어차피 의미 작음.

**UI 영향 (Codex ⚠️ A/D 모호 해소)**:

`Premium.guard === 'warn'` 시 **v0.4.0 의 PremiumGauge `tier-na` 표현 그대로 사용** (premiumPct null + 한글 라벨 "데이터 없음"). 추가 ⚠️ 아이콘이나 별도 "시점 격차 큼" 라벨 **추가 안 함** — 사용자에겐 "데이터 없음" 으로 통일 (스킥, KRX 마감, 환율 없음 등 다른 N/A 케이스와 일관). 단순 + 시각 노이즈 최소.

StockHeroCard.tsx 코드는 이미 `payload.premium?.pctUsd ?? null` 패턴으로 처리 — guard 필드 별도 분기 추가 없음.

### 4.3 부하/차단 모니터링 (Codex 🚨 #2 산식 정정)

**호출량 정량 (산식 정정)**:

- 현재 (v0.4.0, 4s 균일): `3600 / 4 = 900 회/시간 × 24h = 21,600 회/day`
- v0.4.1:
  - 장중 (KRX 09:00-15:30 = 6.5h, 2s TTL): `3600 / 2 × 6.5 = 11,700 회`
  - 휴장 (24h - 6.5h = 17.5h, 7s TTL): `3600 / 7 × 17.5 = 9,000 회`
  - 합계: **~20,700 회/day** (현재 대비 -4% 또는 거의 동일)

→ 일일 총량은 거의 동일하지만 **장중 1시간당 +100%** (900 → 1800). burst 위험 존재.

**Naver 차단 (HTTP 429 / Forbidden) 시 자동 backoff (§10 Open Q-B 결정 옵션 추가)**:
- 기본: sourceHealth 가 자동 stale 마킹 + DegradedBanner 표시 → 사용자 즉시 감지
- 자동 backoff: Naver fetch fail count 3회 연속 시 TTL 자동 7s 로 revert (장중도). 1시간 후 자동 복구 시도 (TTL 2s 재시도).
- 결정: v0.4.1 에선 **수동 revert** (코드 변경, env 토글) — 자동 backoff 는 v0.5.0 backlog. 단순함 우선.
