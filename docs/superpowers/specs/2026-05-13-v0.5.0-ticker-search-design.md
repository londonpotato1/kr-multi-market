## 1. Context & Motivation

v0.4.2 까지 대시보드는 7개 정적 ticker (samsung, skhynix, hyundai, kospi200f, ewy, sp500, nq + usdkrw) 만 표시. 사용자가 임의 종목 (예: AAPL, MSFT, 카카오, NVDA) 모니터하려면 코드 수정 필요.

**사용자 요청**:
> 원하는 ticker 검색하면 가격 모니터 추가 할 수 있도록 하고싶어.
> 신규 티커 검색의 경우 상장되어 있는 거래소를 순차적으로 여러 단계층으로 검색.
> Tier 1: 한국 증권거래소, 미국 나스닥. Tier 2: Binance, OKX 같은 CEX. Tier 3: Hyperliquid.
> 코인 말고 주식에 한해서.

### Goals

1. **검색 input** — 사용자가 종목명 (한글/영문) 또는 ticker symbol 입력
2. **3-Tier resolver** — Tier 1 (KRX/NASDAQ) → Tier 2 (CEX) → Tier 3 (HL xyz) 순차 fallback, 첫 hit 에서 stop
3. **Watchlist 영구 저장** — localStorage 에 사용자 종목 list, 새로고침 후 유지
4. **별도 UI section** — "📌 관심 종목" 별개 grid, 정적 카드와 분리
5. **삭제 기능** — 카드 우상단 X 버튼

### Non-Goals (사용자 명시 + 카파시 결정)

- ❌ 동기화 (multi-device, server stateful) — localStorage only
- ❌ 사용자 인증 / 다중 watchlist
- ❌ 차트, 뉴스, 재무 데이터
- ❌ 알림 (가격 alert)
- ❌ Drag-and-drop 정렬
- ❌ 코인 ticker (BTC/ETH 등) — 사용자 명시 "주식만"
- ❌ OKX, KuCoin 등 추가 CEX — v0.6 backlog
- ❌ 종목명 fuzzy match ("Aple" → AAPL) — exact + prefix only
- ❌ Hyperliquid main dex (crypto) — xyz dex (stock) only
- ❌ FX/Forex ticker (EURUSD)
- ❌ 검색 history, 자동완성 (autocomplete)
- ❌ Watchlist export/import

## 3. Architecture

### 3.1 High-level 구조

```
┌─────────────────────────────────────────────┐
│ Header (코스피 24시간 트래커)               │
├─────────────────────────────────────────────┤
│ [🔍 종목 검색...                    ] [추가]│ ← Q6: SearchBar
│   └─ dropdown (top 5 results) ───────────────┤ ← Q7: SearchDropdown
├─────────────────────────────────────────────┤
│ [메인 한국 주식 3개 카드] (정적, 기존)      │
│ [지수 카드 4개] (정적, 기존)                │
├─────────────────────────────────────────────┤
│ 📌 관심 종목                                │ ← Q8: WatchlistSection
│ [WatchlistCard × N] + 우상단 [X] (Q11)      │
│ (비어있으면 placeholder "검색해서 추가하세요")│
└─────────────────────────────────────────────┘
```

### 3.2 3-Tier Resolver (server-side, `/api/search` 단일 endpoint)

resolver 는 **server-side** 에 위치 (CORS, secret 보호, source 인프라 재사용). client 는 `GET /api/search?q=...` 호출 후 결과 dropdown 표시.

```
입력: GET /api/search?q=AAPL
  │
  ├─ Tier 1: KRX/NASDAQ (외부 search API)
  │   ├─ Hangul detect → searchNaver(q) → KRX 결과 (top 5)
  │   └─ 영문 → searchFinnhub(q) → NASDAQ/NYSE 결과 (top 5, type='Common Stock'|'ETF' filter)
  │   1+ hit → response = { tier: 1, results: [...] } → STOP
  │
  ├─ Tier 2: CEX probe (Binance/Bybit/Bitget) — Tier 1 결과 0개일 때만
  │   ├─ ticker + "USDT" suffix 자동 (AAPL → AAPLUSDT)
  │   ├─ fetchBinanceFutures(['AAPLUSDT']) + fetchBybitLinear(['AAPLUSDT']) + fetchBitgetFutures(['AAPLUSDT']) 병렬
  │   │  (v0.4.2 기존 source 함수 재사용 — 추가 endpoint X)
  │   ├─ 응답 ok && data 가 1+ valid PricePoint → hit
  │   └─ 우선순위 binance > bybit > bitget 으로 first source 채택 → response = { tier: 2, results: [{source, symbol, label}] } → STOP
  │
  └─ Tier 3: HL xyz universe — Tier 2 도 0개일 때
      ├─ getLatestMids() 에서 `xyz:${TICKER}` key 존재 검사 (v0.4.1 hyperliquid-ws.ts)
      └─ hit → response = { tier: 3, results: [{source:'hyperliquid', symbol:`xyz_${TICKER}`, label:TICKER}] } → STOP

모든 tier miss → response = { tier: null, results: [], reason: 'not_found' }
```

**Dropdown 동적 tier**: dropdown 결과 = 현재 hit 된 tier 의 결과만. Tier 1 hit 면 Tier 2/3 미조회. Tier 1 miss 시 Tier 2 결과 (1개) 만 표시. UX 흐름 = Q4 (first-match-wins) 와 일관.

**Frontend `ticker-resolver.ts` 의 책임 축소**: 검색어 → `/api/search` HTTP 호출 + 응답 dropdown 으로 전달만. 3-Tier 분기 server-side.

**입력 정규화** (FLAG fix v3): `q = q.trim()`. 다음 reject:
- 빈 입력 (`q === ""`): client SearchBar `disabled` (Enter 무시) + server `400 Bad Request`.
- 1자 입력: client/server 둘 다 reject ("최소 2자 이상").
- 32자 초과: client/server 둘 다 reject ("32자 이하").
- 허용 문자 외 (특수문자 `;`, `<`, `&` 등): `/^[a-zA-Z0-9가-힣\s.-]+$/` 통과한 것만. 위반 시 reject.

Tier 1 검색 시 Hangul detect = `/[가-힣]/.test(q)` (한글 1자라도 포함 → Naver). Tier 2/3 lookup 전 `TICKER = q.trim().toUpperCase()` (HL `xyz:${TICKER}` + Binance `${TICKER}USDT` 매칭 일관).

**Hangul mixed 입력 처리** (FLAG fix): "SK하이닉스" 같은 한영 mix → Hangul detect 통과 (`/[가-힣]/` 매치) → Naver 검색. Naver 가 한영 mix 검색 지원.

**카파시 가드**: server `/api/search` 안 inline 3-tier (if/else 단일 함수). Tier 별 클래스/Strategy 패턴 X. 단일 endpoint, 단일 함수.

### 3.4 SingleFlight cache key 정책 (BLOCK fix)

검색 probe 와 정기 `/api/prices` poll 의 cache 충돌 방지:

**정기 poll (`/api/prices`)**: 기존 v0.4.2 singleFlight key 유지 — `source:binance`, `source:bybit` 등 (symbols 무관, source 별 단일 in-flight). symbols 는 정기 poll 마다 동일 set (정적 + 현재 watchlist 합집합) 이므로 같은 key 안전.

**검색 probe (`/api/search` 안 Tier 2 CEX 호출)**: **singleFlight 우회**. `fetchBinanceFutures(['AAPLUSDT'])` 등을 직접 호출 (cache.ts 미경유). 검색 호출 빈도 = 사용자 검색 시 1회 (10초당 < 1회 추정). 정기 poll cache (1s TTL) 와 분리.

```ts
// 정기 poll (server/index.ts) — 기존 v0.4.2 패턴 유지
singleFlight('source:binance', SOURCE_TTL_MS.binance, fetchBinanceFutures);

// 검색 probe (server/routes/search.ts Tier 2) — singleFlight 우회
const result = await fetchBinanceFutures([`${TICKER}USDT`]);
```

**근거**:
- 정기 poll = 모든 watchlist + 정적 symbols 의 batch 호출. singleFlight 가 중복 호출 흡수.
- 검색 probe = 사용자 검색 1회. singleFlight 쓰면 cache key 가 정기 poll 과 충돌 → 잘못된 batch 반환. 분리가 안전.
- 비용: 검색 시 Tier 2 probe = Binance + Bybit + Bitget **3 source 병렬 호출** (각 1회). rate limit 무시 가능 (사용자 검색 = 10s 당 1회 미만 추정).

### 3.3 Data Flow

#### 검색 → 추가
```
SearchBar Enter
  → GET /api/search?q=Apple
  → [server] Tier 1 (Hangul detect → Naver / 영문 → Finnhub /search)
     → Tier 1 miss 시 Tier 2 (CEX probe via 기존 fetchX 함수)
     → Tier 2 miss 시 Tier 3 (HL WS state lookup)
  → response { tier, results }
  → SearchDropdown 표시 (top 5 or 1)
  → 사용자 클릭 → entry = results[i]
  → useWatchlist.add(entry)
  → localStorage 갱신 + SWR mutate
  → WatchlistSection 카드 즉시 렌더 (loading)
  → 다음 SWR poll (2s) 후 가격 표시
```

#### 가격 fetch (매 2s SWR poll)
```
useWatchlist.entries → SWR key 구성
  → GET /api/prices?watchlist=aapl:finnhub:AAPL,kakao:naver:035720,tsla-perp:binance:TSLAUSDT
  → server: parseWatchlist → [{key, source, symbol}, ...]
  → 동적 source-별 symbols 합집합 (기존 정적 + 동적 watchlist)
  → source-level singleFlight cache 흡수 (정기 poll 만, key 정책 §3.4)
  → 정적 SYMBOL_TO_TICKER + 동적 watchlist mapping 통합 (entry.key 로 tickers entry 추가)
  → tickers 응답 (정적 + 동적)
  → WatchlistCard 가 IndexCompactCard 패턴으로 표시
```

**Query encoding** (BLOCK fix v3 — delimiter safety 명시):

`watchlist=KEY:SOURCE:SYMBOL,...` 형식. 각 field 의 **허용 문자 집합 명시**:
- `KEY` = `/^[a-z0-9][a-z0-9-]*$/` (lowercase alphanumeric + hyphen, 최대 32자). `:` `,` 절대 포함 X.
- `SOURCE` = `SourceName` 9 keys 중 하나 (`hyperliquid|naver|yahoo|binance|upbit|bybit|bitget|polygon|twelvedata`). 고정 문자열만.
- `SYMBOL` = `/^[A-Z0-9_]+$/` (uppercase alphanumeric + underscore, 최대 32자). KRX 6자리 숫자 (`005930`), 영문 ticker (`AAPL`, `QQQUSDT`), HL prefix (`xyz_KR200`) 모두 포함. `:` `,` 절대 포함 X.

**percent-encoding 사용 X** — 모든 field 가 alphanumeric + `-`/`_` 만 → URL-safe. delimiter `:`/`,` 와 충돌 0.

**검증 위치**:
- `useWatchlist.add()` 가 entry 추가 전 KEY/SYMBOL regex 검증. 위반 시 reject + alert.
- server `parseWatchlist()` 가 query 파싱 시 동일 regex 재검증. 위반 시 invalid entry skip.

예: 3 종목 watchlist = `aapl:finnhub:AAPL,kakao:naver:035720,tsla-perp:binance:TSLAUSDT` ≈ 70 bytes. 50 종목 = ~3.5KB (URL 한도 8KB 안전, percent-encoding 없으므로 팽창 X).

**LABEL + TIER 는 query 제외** — client localStorage 에 보관, server 응답에 불필요.

#### 삭제
```
WatchlistCard 우상단 X 클릭
  → useWatchlist.remove(key)
  → localStorage 갱신 + SWR mutate
  → 카드 즉시 사라짐
  → 다음 SWR poll 부터 server query 에서 제외
```

## 5. localStorage Schema

**Key**: `kr-multi-market.watchlist.v1`

```json
[
  { "key": "aapl", "source": "finnhub", "symbol": "AAPL", "label": "Apple Inc.", "tier": 1 },
  { "key": "kakao", "source": "naver", "symbol": "035720", "label": "카카오", "tier": 1 },
  { "key": "tsla-perp", "source": "binance", "symbol": "TSLAUSDT", "label": "Tesla (Binance)", "tier": 2 }
]
```

### Key 충돌 처리

- 사용자가 정적 ticker (예: "EWY", "QQQ", "삼성전자") 검색 시 → 기존 정적 카드 있으니 **alert + 거부** ("이미 표시 중").
- 검색 key 가 다른 watchlist entry 와 충돌 시 → `-1`, `-2` suffix 자동 추가.

### 용량 제한

- 최대 50 entries (사용자 alert)
- localStorage 5MB cap 안전 마진
- **Enforcement 위치**: client-side only (useWatchlist.add 가 50 초과 시 reject). server 는 받은 watchlist query 그대로 처리 (soft cap, server 강제 X) — server stateless 유지.

### 멀티탭 동시 추가 (FLAG fix)

같은 origin 의 다중 브라우저 탭이 동시에 종목 추가 시:
- **Last-write-wins**: 마지막 `localStorage.setItem` 호출이 이전 탭 변경 덮어씀.
- 다른 탭 `storage` 이벤트 listen 안 함 (v0.5.0 scope 외, v0.6 backlog).
- 사용자 영향: 두 탭 동시 추가 시 한 entry 사라질 수 있음. 새로고침으로 복원.

## 6. Error Handling + Edge Cases

| 시나리오 | 처리 |
|---|---|
| 빈 입력 / whitespace only | SearchBar Enter disabled (button 비활성), server 호출 X |
| 1자 입력 / 32자 초과 | dropdown "검색어 2-32자 입력" toast |
| 특수문자 (`;`, `<`, `&` 등) | dropdown "허용되지 않는 문자" toast |
| 검색 결과 0개 | dropdown "검색 결과 없음" 표시 |
| Finnhub 60/min quota 초과 (429) | "잠시 후 다시 시도" + 60s cooldown timer |
| Naver scrape 실패 (한글 입력) | dropdown "한국 주식 검색 일시 장애. 영문 ticker (예: 005930) 로 직접 입력하세요" 메시지. Finnhub 는 한글 검색 미지원 → 영문 fallback 없음. |
| Naver scrape 실패 (영문 입력) | 해당 없음 (영문은 처음부터 Finnhub 사용) |
| 모든 Tier miss | dropdown "$ticker 어느 거래소에도 없음" + 추가 거부 |
| watchlist 중복 추가 (같은 key) | alert "이미 추가됨" + 거부 |
| 정적 ticker 와 충돌 | alert "이미 표시 중 (메인 카드)" + 거부 |
| localStorage 용량 초과 (5MB or 50 종목) | warning + 추가 거부 |
| server `/api/prices` watchlist source fail | 카드 표시 + 가격 "—" + sourceHealth 마킹 |
| 검색 ticker 가 코인 ("BTC", "ETH") | Finnhub filter (Common Stock + ETF type only) → dropdown 결과 없음 |
| localStorage 손상 / JSON parse fail | try/catch → 비우고 경고 toast |
| 새로고침 후 watchlist 복원 | useWatchlist init 시 localStorage 읽기 → SWR 자동 fetch |
| HL universe 확장 (새 ticker) | WS allMids 자동 catch → Tier 3 lookup 가능 |
| 모바일 < 640px | SearchBar full-width + 1-column watchlist grid |

## 10. Risks

| # | Risk | Mitigation |
|---|---|---|
| R1 | Finnhub /search 60/min — 검색 자주 시 quota 소진 | Enter submit only (Q10) + 60s cooldown |
| R2 | Naver search endpoint 비공식 (undocumented JSON) | 응답 shape 검증. 한글 입력 시 fail → 사용자에게 "영문 ticker (예: 005930) 직접 입력" 권고 메시지. **Finnhub fallback 안 함** (한글 검색 미지원). §6 일관성. |
| R3 | watchlist 50 종목 시 SWR query string ~3.5KB (key:source:symbol triple, BLOCK fix encoding 기준) | URL 한도 8KB 안전 마진 충분 |
| R4 | HL xyz universe 의 ticker prefix 모호 ("NDX" vs "^NDX") | `xyz:` prefix 만 lookup |
| R5 | 코인 ticker filter 누락 — Finnhub `type` 필드 없을 때 | filter 미적용 시 통과 (BTC 가 Common Stock 분류 X 라 자연 제외) |
| R6 | unit drift (AAPL Binance perp ≠ NASDAQ ETF) | Tier 1 우선 (Q4) — NASDAQ 가격 채택, 1 venue 카드라 drift visible X |
| R7 | localStorage 손상 시 watchlist reset | try/catch + corrupt 시 비우고 경고 |
| R8 | Tier 2 probe (CEX) 시 3 source 병렬 호출 매번 — quota | 검색 시 1회만 호출, 결과는 localStorage 영구 캐시 |
| R9 | 검색 probe singleFlight 우회 (§3.4) — 정기 poll 과 cache 분리. Tier 2 probe = Binance + Bybit + Bitget **3 source 병렬 호출** (1회씩, 총 3 API call) | 검색은 사용자 trigger 1회 — Binance 6000 weight/min + Bybit 60 req/s + Bitget 20 req/s 대비 부담 무시 |
